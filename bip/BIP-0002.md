## Abstract

This BIP ratifies Biomarker Taxonomy v1 as the canonical biomarker reference for BSP. The taxonomy defines 210+ biomarker codes across four levels — Core (longevity), Standard (routine laboratory), Extended (specialized), and Device (continuous wearable) — with standardized code format, units, reference ranges, and evidence grading. The taxonomy is the universal language that lets any lab, wearable, clinic, or AI system describe biomarker values without ambiguity.

---

## Motivation

Current health data interoperability standards (LOINC, SNOMED, UCUM) were designed for billing and clinical documentation. They are incomplete for:

- Longevity biomarkers (GDF-11, TIMP2, DNAm age).
- Continuous wearable signals (HRV variants, sleep architecture).
- Advanced immunology and mitochondrial markers.
- Polygenic risk scores and multi-omic derived metrics.

BSP needs a taxonomy that is:

- **Complete** for the use cases BSP targets (longevity, precision health, AI-driven diagnostics).
- **Unambiguous** — every code maps to exactly one biomarker, unit, and measurement method.
- **Machine-readable** — structured YAML with a meta-schema.
- **Evolvable** — new codes can be added via BIP-T without breaking existing implementations.
- **Interoperable** — every code carries an optional LOINC / UCUM mapping when one exists.

Taxonomy v1 ships with the protocol and becomes the reference that all v1.0 BEOs, IEOs, and BioRecords must resolve against.

## Specification

### Code format

```
BSP-[CATEGORY]-[NUMBER]
```

- `CATEGORY`: 2–3 uppercase letters identifying the biomarker family (e.g., `GL`, `LA`, `DV`).
- `NUMBER`: zero-padded 3-digit sequence within the category (e.g., `001`).

Examples: `BSP-GL-001` (Fasting Glucose), `BSP-LA-004` (NAD+), `BSP-DV-001` (HRV RMSSD).

### Levels

| Level | Purpose | Count |
|---|---|---|
| CORE | Advanced longevity biomarkers | 9 categories |
| STANDARD | Routine laboratory | 9 categories |
| EXTENDED | Specialized assays | 6 categories |
| DEVICE | Continuous wearable signals | 1 category |

Totals as of v1: 210+ individual biomarker codes across 25 categories.

### Categories — Level 1 Core

- `BSP-LA` — Longevity & Aging
- `BSP-RC` — Regeneration & Cellular
- `BSP-CV` — Cardiovascular Health
- `BSP-IM` — Immune & Inflammation
- `BSP-ME` — Metabolism & Energy
- `BSP-NR` — Neurological Health
- `BSP-DH` — Detoxification & Hepatic
- `BSP-LF` — Lymphatic & Clearance
- `BSP-BC` — Biological Clock

### Categories — Level 2 Standard

- `BSP-HM`, `BSP-VT`, `BSP-MN`, `BSP-HR`, `BSP-RN`, `BSP-LP`, `BSP-GL`, `BSP-LV`, `BSP-IF`

### Categories — Level 3 Extended

- `BSP-FR`, `BSP-GN`, `BSP-MB`, `BSP-TX`, `BSP-IM2`, `BSP-CV2`

### Categories — Level 4 Device

- `BSP-DV`

### Source of truth

Full taxonomy lives under [`spec/taxonomy/`](../spec/taxonomy/):

- `spec/taxonomy/level-1-core/*.yaml`
- `spec/taxonomy/level-2-standard/*.yaml`
- `spec/taxonomy/level-3-extended/*.yaml`
- `spec/taxonomy/level-4-device/*.yaml`

Each biomarker YAML file contains: code, name, display_name, category, level, unit, method, reference ranges (optimal / functional / deficiency / toxicity), cost tier, and evidence.

### Meta-schema

Every taxonomy file must validate against `spec/taxonomy/_meta-schema.yaml`. CI enforces this on every PR.

### Reference implementation

All BSP SDKs expose a `TaxonomyResolver`:

```typescript
// TypeScript SDK
import { TaxonomyResolver } from '@bsp/sdk'

const resolver = new TaxonomyResolver()
const biomarker = await resolver.get('BSP-GL-001')
// => { code: 'BSP-GL-001', name: 'Fasting Glucose', unit: 'mg/dL', level: 'STANDARD', ... }
```

```python
# Python SDK
from bsp_sdk import TaxonomyResolver

resolver = TaxonomyResolver()
biomarker = resolver.get("BSP-GL-001")
```

The resolver loads from the local taxonomy bundle shipped with the SDK. Updates arrive through SDK releases pinned to taxonomy version.

## Rationale

### Why a BSP-prefixed code and not reuse LOINC directly?

LOINC does not cover the longevity and continuous-device space in enough depth. BSP codes reference LOINC where a mapping exists (see each YAML's optional `mappings.loinc` field) and extend coverage where LOINC is silent.

### Why four levels instead of a flat list?

Levels enable:

- Cost-aware client UIs (show only STANDARD by default).
- Device integration scope (a wearable ships with L4 codes only).
- Certification tiers for IEOs (basic IEOs certify on STANDARD; longevity IEOs on CORE + EXTENDED).

### Why YAML and not JSON?

Taxonomy files are authored by humans (biologists, clinicians). YAML is easier to review. Machine consumers use pre-built JSON bundles generated at release.

### Alternatives considered

| Alternative | Reason rejected |
|---|---|
| Reuse LOINC as-is | Insufficient coverage for longevity and wearable domains. |
| Adopt SNOMED-CT codes | Licensing restrictions incompatible with BSP's MIT license. |
| Flat numeric code space | Loses semantic grouping; harder for humans to reason about. |
| Inline taxonomy in SDK source | Couples taxonomy version to SDK version; blocks rapid additions via BIP-T. |

## Backward Compatibility

- [x] This change is fully backward compatible

BSP v1.0 is the first version. No prior taxonomy exists.

For future taxonomy updates: code reuse is prohibited. Deprecated codes are kept with `status: DEPRECATED` and a pointer to the successor. Clients must tolerate unknown codes gracefully (preserve but flag).
